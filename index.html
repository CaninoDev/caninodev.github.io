A Stately Examination

A Stately Examination on the Lifecycle of React Component

Introduction

Recently was working on a project and came across a situation where some pre-render manipulation of props were required. Ordinarily this would be handled via componentWillReceiveProps. However the React team decided to deprecate this function and, in its place, for very rare circumstances, getDerivedStatesFromProps was provided in React 16. However there is some confusion and concerns as to how it was meant to be used. The following is a concrete use case on the usage of this functionality, and in addition to the guidelines and examples provided by the team, may prove helpful in further understanding.

Scenario

In my project, I wanted to render a chart that would provide certain aggregate calculations on two data points. However, one data point was dependant on user input and, ideally, the chart should render the other data point. The situation was further complicated by a specific data format that must be fed into the React component rendering the chart. In this particular project, Nivo‘s responsive radar chart is the component in question. The syntax for this component is roughly as followed:

Deriving State


As stated by the react team:

[…] as a general rule, derived state should be used sparingly.

  1. If you’re using derived state to memoize some computation based only on the current props, you don’t need derived state.
  2. If you’re updating derived state unconditionally or updating it whenever props and state don’t match, your component likely resets its state too frequently.

In this scenario, the state is depended on the source props and will always change when the props are updated with new data.
!!!note getDerivedStateFromProps is a static method
Unlike componentWillReceiveProps, this is not available in getDerivedStateFromProps.

Kitting Out With Homekit/HomeBridge

Kitting out the home with Homekit/Homebridge (part I)

After having spent two months attending the intense and illustrious Flatiron School, I decided to get my hands dirty with a hands on side-project at home. I wanted to hone my skills on something that was somewhat practical and would present enough new concepts to be challenging but not overwhelming for my skill level.

Since it is the heart of summer, what better project to dive into than to integrate my 7 year old LG LW1010ER Air Conditioner controlled by a basic remote with Homekit.

!!!note Note
I chose Homekit as I already have bought into the Apple ecosystem with several iOS devices and prefer the available integration the API framework provides. Furthermore, the framework is fairly basic and is ideal for an advanced beginner in Javascript.

There are a number of other frameworks that one may want to consider (i.e. Domoticz, Home Assistant, and Calaos) but are outside the scope of this article.
!!!

Considerations

This article assumes a solid understanding of ES6+ Javascript and an ability to control an IR transmitter programmatically.

I already have a Raspberry Pi (currently used as a Pi-hole® ad-blocking DNS and PiVPN server). Since the integrated GPIO pins are not being used, I purchased an IR Infrared Remote Control Transceiver Shield and installed LIRC, a library framework to control the IR shield. For more information on setting up LIRC, refer to setting up a remote control using lirc.

A javascript implementation that emulates Apple’s HomeKit API called Homebridge will be used in this case-study.

What is Homebridge?

From their github:

Homebridge is a lightweight NodeJS server you can run on your home network that emulates the iOS HomeKit API. It supports Plugins, which are community-contributed modules that provide a basic bridge from HomeKit to various 3rd-party APIs provided by manufacturers of “smart home” devices.

Apple’s HomeKit Developer Guide is a great resource to learning about Homekit but is a bit overwhelming. Homebridge and its underlying engine, HAP-NodeJS, handles much of the heavy lifting for you. It is a great starting point and leaves the developer just with defining the accessories and coding the conotrol logic. A plethora of plugins have already been written and thus provides many examples to study from to gain a better understanding of the framework.

Terminology

Bridge

A bridge is a special type of occessory (see below) that facilitates communication for accessories that does not natively support the Homekit framework or Homekit Accessory Protocol (HAP). Accessories that rely on IR transmission, Wink‘s protocol, or, if you want to be really creative, a pattern of knocks for control can be translated by the bridge to HAP. It facilitates communication on a device’s attributes, status and translates control commands in the accessory’s native format. In order to take advantage of many of Homekit’s automation features, the bridge should be installed on an always-on network-connected device. In this example, Hombridge will act as a bridge and is installed on a Raspberry Pi.

Accessory

An accessory is the physical device in question such as a light, garage door, thermostat, etc… An accessory has one or more services associated with it.

Service

A service represents the function of the accessory and consists of one or more characteristics. A service may be a light service to control on/off and dimmer states of a device or a thermostat service to control temperature.

Characteristic

A characteristic is a property of a service such as current state of the on/off switch, the current temperature, or target mode of the device. A characteristic may be either read-only or read-write.

!!!note Note
While it is possible to develop custom services and characteristics for a device, Siri only recognizes Homekit’s predefined services and characteristics. HAP-Nodejs’s supported types provides a list of such services and characteristics recognizable by Homekit through Homebridge. Towards the bottom of this source code is a list of services along with their required and optional characteristics. As of this writing, there is a 1:1 parity between Homekit and HAP-nodejs.

It is recommended to use a combination of pre-defined services and characteristics to mirror an accessory’s capability rather than developing custom service(s). This is done by linking services to a primary service. This will be expanded on later in this case-study.
!!!

Formal Language

(Casual Babblings)

To first understand regular expressions, we must examine what a formal language is. Gaining this understanding will help clear away the apparent obfuscation of its syntax. According to wikipedia, a formal language is

is a set of strings of symbols together with a set of rules that are specific to it.

The following is such a language:

(String of symbols)

  • ∅ denoting the set ∅.
  • ε denoting the set containing only the “empty” string, which has no characters at all.
Wait wait wait… Are we learning greek?

no..

  • a in Σ denoting the set containing only the character a.

(Operators)

  • (concatenation) let R is “Did you hear about the restaurant on the moon?” and S is “Great food, no atmosphere”, thus RS is “A terrible joke”
  • (alternation) let R be {“ab”, “c”} and let S ne {“ab”. “d”, “ef”}, thus R | S is {“ab”,”c”,”d”,”ef”}
  • (Kleene star) R* is the smallest superset of sets that contains ε and contains all combinations of strings that can be concatanated. For example. {“ab”, “c”}* = {ε, “ab”, “c”, “abab”, “abc”, “cab”, “cc”, “ababab”, “abcab”, … }.

If you would like to know more about Formal Languages, read up on Chomsky hierarchy. In fact, all programming langauges are considered to be a Formal Language. So you are already familiar with them. Regular expressions are no different except that its syntax consists of various symbols with no spaces.

Typical Use Cases

(Including Strange and Unusual Ones)



Regular expressions (regex) are extremely powerful. They can be used to match telephone numbers in its various formats: (xxx)-yyy-zzzz, xxx-yyy-zzzz, xxx.yyy.zzzz. It can be used to find a substring ‘needle’ that is only seven words away from ‘haystack’ and is the first word of a sentence. Its used by the syntax highlighting engine of your editor. It’s even used in ezGestures to interpret gestures that are represented as a series of UDLR strings. These are some of the more pragmatic use cases for regular expressions:

  • Parsing and validating user input (password)
  • Restructuring user input (date formatting, currency)
  • Converting system logs for use with a plotting to graph performance
  • Refactoring code via search and replace
  • Getting a trailing folder from a path

Another great thing about regex is that most languages, including Ruby, can interpret regular expressions. However check with your language reference manual to see if there are any variance in syntax.

And without further ado, here is some of the syntax, keeping in mind that the operator acts on what comes before it:

However…

Some people, when confronted with a problem, think “I know, I’ll use regular expressions.” Now they have two problems. –Jamie Zawinski

Even veteran developers who frequently use regex in their every day tasks for 10+ years still have to refer to cheat sheet. While regex’s are powerful, use them prudently. Sometimes there is an alternative tool that can get tbe job done faster than the time it takes to figure out the proper expression. The following are some use cases where regex’s are not recommended:

  • Parsing HTML with regex
  • Parsing natural langauges, for example obscenity/blacklisted words
  • Despite the example in the beginning of this presentation, validating email addresses.

A Bug's Life

Is it possible to produce bug-free software?

 - -

A Bug’s Life

 - - - - - -

“We study the past to understand the present; we understand the present to guide the future.” - William Lund

 - - - - - - - - - - - -

“Moth found trapped between points at Relay # 70, Panel F, of the Mark II Aiken Relay Calculator while it was being tested at Harvard University, 9 September 1947. Operators affixed the moth to the computer log, with the entry: “First actual case of bug being found”. Grace Hopper publicised this incident

I had believed for the longest time that this woman, Rear Admiral Grace Hopper, first coined the term “bug” and “debugging”.

A native New Yorker, she became the first woman to earn a Ph.D in mathematics from Yale University in 1934. She eventually went on to consult in development of COBOL (a language still widely used today!), had a US Navy ship christened in her name, a college in Yale named after her, won 40 honorary degrees from universities around the world, and posthumously awarded the Presidential Medal of Freedom by President Obama.

While during research for this blog posting, to my chagrin, this turns out not to be the case. This doesn’t detract from her all around badassery however. I highly encourage learning about her. She significantly accelerated the development of programming languages with her significant contribution to COBOL (a langugage still used today!), been described “as appearing to be all Navy, but when you reach inside, you find a ‘Pirate’ dying to be released”, and famously said: “It’s easier to ask forgiveness than it is to get permission. (and its variant: “If it’s a good idea, go ahead and do it. It is much easier to apologize than it is to get permission.”).

“If names be not correct, language is not in accordance with the truth of things.” - Confucius, The Analects of Confucius
In fact, the term ‘bugs’ and ‘debugging’ were commonly used at the time among engineering circles. In fact, originally bugge meant “something frightening; scarecrow.” and only just transformed its meaning during the 1870’s. Thomas Edison (NIKOLA TESLA RULES!) wrote:
It has been just so in all of my inventions. The first step is an intuition, and comes with a burst, then difficulties arise - this thing gives out and [it is] then that “Bugs” - as such little faults and difficulties are called - show themselves and months of intense watching, study and labor are requisite before commercial success or failure is certainly reached.
Among certain, - -ahem - - hipster engineering circles, ‘bugs’ are further redefined as ‘mistake metamorphisms’.
Software bugs are the cause of the famous Mars Climate Orbiter‘s disintegration in Mar’s atmosphere (we really all should just get on the metric unit), the destruction of the alien mothership in Independence Day, and most importantly (and my personal favorite) a bug in Sid Meier’s Civilization that would cause the normally peaceful Ghandhi to become the most agressive civilization leader in the game.
After scouring Stack Exchange (and their numerous forums), industry articles covering computer science, the consensus is that beyond simple Hello_World programs, software bugs are pretty much a certainty. (Except when it is a feature.)

 - - - - - -

Ant Colony

 - - - - - 
Delving deeper into the rabbit-hole (andf mixing up my metaphors), I wanted to know why. It came down to the principle of Emergence. Many of the postings and articles I have read eventually pointed to Conway’s Game of Life.

This game has 4 basic rules:

  1. Any live cell with fewer than two live neighbors dies, as if caused by under population.
  2. Any live cell with two or three live neighbors lives on to the next generation.
  3. Any live cell with more than three live neighbors dies, as if by overpopulation.
  4. Any dead cell with exactly three live neighbors becomes a live cell, as if by reproduction.

    Depending on starting conditions, the system has properties that its parts do not have. This game is in a perfect system, thus unanticipated patterns repeat indefinitely.
    However, software never exists in isolation and do not always produce intended behavior. It is a part of a digital ecology of imperfect actors with partially realized intentions. Interactions are thus unpredictable and sometime with drastic outcomes. These bugs pupates to produce Butterfly Effects
